# 实验方案：面向长文本复杂 SQL 推理的端到端数据流优化与快进引擎

## 一、 核心业务场景与端到端推理走势 (The Context & Flow)

本系统专为**企业级超长 OLAP SQL / ETL 脚本的自动化生成与补全**场景设计。

典型的一条请求包含：*巨型的基础表 DDL、复杂的业务元数据描述、多层嵌套的子查询，以及前后一字不差、高密度绝对重复的复杂计算逻辑*（例如 `SELECT` 门类和 `GROUP BY` / `ORDER BY` 门类后面长达数百行的 `CASE WHEN` 条件映射链）。

从端到端推理的角度来看，一条请求的完整生命周期会经历三个核心阶段：**Prefill前置期 $\rightarrow$ Prefill（预填）期 $\rightarrow$ Decode（自回归生成）期**。我们必须在这三个阶段的数据流节点上进行针对性拦截。

------

## 二、 端到端数据流拦截方案与“为什么这样做” (End-to-End Dataflow Design)

### 阶段 1：Prefill 前置期 —— 从“文本”到“结构感知 Token 数组”

- **数据流节点**：用户长提示词（Prompt）到达，经过 Tokenizer 变成一维 Token ID 数组。

- **面临瓶颈**：复杂的 SQL 提示词中夾杂着大量长代码注释、大段静态 Mock 数据列表。如果让这些“非逻辑、纯数据”的 Token 序列直接进入 NPU 进行 Prefill 计算，会迅速吃满 KV Cache 显存并拖慢计算时间。此外，中段由于局部绝对重复（如多处出现的相同 `CASE WHEN` 文本），框架此时还一无所知。

- **我们为什么要做前置准备？**

  在大规模矩阵乘法开始前，必须由 CPU 侧进行轻量级的预解析（Pre-parsing），为后续的 NPU 内存分配和跳跃计算生成“引导说明书”，避免 NPU 在计算时频繁产生控制流中断。

- **Claude Code 落地方向**：

  在 Token 数组生成后、进入推理队列前，引入拦截器执行两个动作：

  1. **字面量动态压缩**：识别连续长注释或长列表，在 Token 数组层原地“塌陷”替换为特殊锚点 Token（如 `[Data_Block_Start] -> [Length] -> [Data_Block_End]`），直接缩短进入 Prefill 的原始数组长度。
  2. **跨位置重复静态打标**：对局部复杂的逻辑块（如 `SELECT` 后的 `CASE WHEN`）进行滑动窗口 Hash 签名，并计算其 Token 相对偏移量，生成一份 `AliasingGuideTable`（流内复用引导表）绑在请求上。

------

### 阶段 2：Prefill 期 —— 从“计算 KV”到“页表别名化拓扑”

- **数据流节点**：压缩后的 Token 数组送入 NPU 执行前向传导，计算注意力（Attention）矩阵并生成最初的 KV Cache 物理页。

- **面临瓶颈**：当 Prefill 流处理到 SQL 中段时，传统的 RadixAttention 只能识别请求“开头”的公共前缀复用（Prefix Caching）。由于 `SELECT` 后的 `CASE WHEN` 并不在开头，系统会盲目地为它重复计算 KV Cache 并且申请全新的物理显存页，导致显存随长文本呈恶性线性飙升。

- **我们为什么要做页表别名化（Page Aliasing）？**

  既然这段代码逻辑在当前查询中是 100% 绝对重复且静态的，那么在物理存储上，它**理应只占用一份空间**。我们需要打破“一个虚拟 Token 序列对应一组新物理页”的传统假设，允许系统进行跨位置的内容寻址。

- **Claude Code 落地方向**：

  魔改 Mini-SGLang 的内存管理器（`mem_cache.py` / `RadixCache`）：

  - 当 Prefill 扫描到中段，触发前置期生成的 `AliasingGuideTable` 时，页表管理器（Page Table）直接介入拦截。
  - 停止为这一段重复的 Token 序列向显存申请新物理页，而是**直接把当前的虚拟页号（Virtual Page Number）别名化，将其指针强行重定向到前半段已经计算完毕的相同逻辑块的物理页号（Physical Page Number）上**。实现显存的“原地瘦身”。

------

### 阶段 3：Decode 期 —— 从“逐 Token 过滤”到“内核级快进缝合”

- **数据流节点**：模型开始自回归（Autoregressive）逐个预测下一个 Token，直到拼装出完整的 `GROUP BY` 及其后面的复杂字段。

- **面临瓶颈**：

  1. **模型端**：长上下文下注意力会衰减，模型写到后半段 `GROUP BY` 时会发生健忘，导致写错或无法与 `SELECT` 的字段对齐。
  2. **工程端**：为了让它对齐，传统的“语法引导解码（Constrained Decoding）”是在 Python 层用状态机或正则拦截，这导致每个 Token 都要进行 CPU-NPU 回传，产生巨大的流水线空泡，推理速度爆炸慢。

- **我们为什么要做动态快进缝合（Flash-Forward Decoding）？**

  既然语法状态机（FSM）已经明确判定当前必须写 `GROUP BY` 后的内容，且通过 Hash 树判定这个内容必然是一字不差地复制前面的 `SELECT` 块，**那大模型在这个区间内就完全没有进行“自回归概率预测”的必要**。直接用底层的物理内存块把它“缝合快进”过去，既保证了 100% 对齐，又免去了多步 NPU 前向计算。

- **Claude Code 落地方向**：

  1. **语法状态机位图下沉**：离线将 SQL 语法编译为有限状态机（DFA），并与模型词表（Vocab）映射为静态位图（Bitmask），将其常驻在采样算子层。推理时通过 $O(1)$ 查表直接干预 NPU 侧的 Logits 概率分布，干掉 Python 层高频同步。
  2. **内核级跨模块快进（Flash-Forward）**：重构重叠调度器（`scheduler.py`）。当状态机判定进入 `GROUP BY` 且命中流内 Hash 重复块时，调度器直接中断当前 Sequence 的标准自回归流，**在 Runtime 层面将前文 `SELECT` 块的 Token 数组以及对应的物理页指针直接追加缝合到当前的页表尾部**。完成“物理快进”后，直接将序列步数（Step）推向结束点，规避这数百个 Token 的自回归 Forward 计算。

## 三、 实验评测方案 (Evaluation Setup)

为了让 Claude Code 能够闭环验证它写出的代码是否有性能和准确度增益，我们构建以下评测链路：

- **基础底座**：`Qwen/Qwen2.5-Coder-7B-Instruct`（或 32B 版本）。
- **测试数据集**：构造/收集 100 条企业级复杂 OLAP SQL 生成 Prompts。确保每条 SQL 包含多层嵌套、巨型 `CASE WHEN`，单次查询内绝对重复的代码块长度在 200 ~ 1000 Token 之间，整体上下文长度梯度覆盖 8K、32K、128K。
- **对比基线 (Baselines)**：
  1. 原始原生 `Mini-SGLang`（默认开启线性 Prefix Caching）。
  2. 原始原生 `Mini-SGLang` + 外挂式 Python 层语法约束（模拟 Outlines 逐 Token 拦截行为）。

------

## 四、 核心评估指标 (Metrics)

Claude Code 完成重构后，系统需要跑通并量化以下三个硬核指标：

1. **语法绝对对齐率 (Accuracy)**：
   - *指标定义*：`SELECT` 与 `GROUP BY` 内部字段与条件链的完全对齐率（100% Exact Match）。
   - *预期收益*：魔改系统必须通过物理硬干预达到 100% 绝对对齐，彻底干掉原生自回归的位置衰减幻觉。
2. **显存释放红利 (Memory Footprint)**：
   - *指标定义*：Peak Memory Usage（峰值显存）及随着上下文拉长的显存增长斜率。
   - *预期收益*：在生成包含巨型重复逻辑的代码时，**动态 KV Cache 显存开销相比基线直接砍掉 40% - 60%**，在重复块区间显存曲线呈现完美的平台期（零增量）。
3. **端到端吞吐率提升 (Throughput & Latency)**：
   - *指标定义*：TTFT（首 Token 延迟）、ITI（Token 间生成延迟）、每秒处理 Token 数（QPS）。
   - *预期收益*：不仅彻底消除了传统外挂约束带来的延迟飙升，且由于 `Flash-Forward` 机制直接跳过了大量 Token 的模型前向计算，**端到端生成吞吐量（QPS）应反超原生无约束推理 1.5 倍以上**。
